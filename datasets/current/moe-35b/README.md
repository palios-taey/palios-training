# Training data — the actual set that produced the Taey 35B-A3B model

This is the **real training data**, in the format it was ingested, for the production Taey model: `phase_combined_v1` (SFT baseline, 82.8%) → `religion_dpo_v2` (Config A2 DPO, 84.7%). Released so the pipeline can be replicated end-to-end, not just read.

| File | Records | Phase | Format |
|---|---|---|---|
| `combined_v1_gated.jsonl` | 2,325 | SFT (`SFT_DIR`) | Seed-42 combination of the two gated SFT inputs |
| `constitutional_gated.jsonl` | 1,378 | SFT input | `{"messages":[{"role":"user",...},{"role":"assistant",...}]}` |
| `phase1_infra_v2_gated.jsonl` | 947 | SFT input | same |
| `religion_v3_dpo_pairs.jsonl` | 50 | DPO (`DPO_DATA`) | `{"prompt":..., "chosen":..., "rejected":...}` (chosen = authored; rejected = abliterated-base output) |

The production SFT launcher consumes `combined_v1_gated.jsonl` directly (`SFT_DIR` points here and `SFT_GLOB=combined_v1_gated.jsonl`). The two input files are shipped so [`../../../scripts/combine_phase_combined_v1.py`](../../../scripts/combine_phase_combined_v1.py) can regenerate the public combined corpus with `--seed 42`. `DPO_DATA` points at the pairs file — the production run used a `_with_ref` variant that adds precomputed reference log-probs.

Public corpus reproducibility:

```bash
python3 ../../../scripts/combine_phase_combined_v1.py \
  --identity constitutional_gated.jsonl \
  --infra phase1_infra_v2_gated.jsonl \
  --output /tmp/combined_v1_gated.jsonl \
  --seed 42
sha256sum /tmp/combined_v1_gated.jsonl
```

## How it was built — synthetic, not transcripts

The SFT data was generated by sending source documents to an AI and having it author Q&A/instruction pairs about them ("Bot sends doc as attachment → AI generates pairs"); the DPO pairs are authored `chosen` vs abliterated-base `rejected`. **There are no conversation transcripts** — no captured human↔AI chat logs, no session dumps. Every record is authored-for-training on top of source material.

## What the source material is

- **First-party doctrine & identity material** — the project's own identity and alignment source documents (`kernel/`, `layer_0/`, `layer_1/`).
- **Technical / public-domain knowledge** — e.g. the CUDA programming guide, arxiv papers (Switch Transformer, attention, abliteration), framework docs. These carry their own authors' published bylines/contact emails and vendor support addresses (public, not private — see `REDACTIONS.md`).
- **`layer_2` (personal/identity tier)** — additional first-party identity material the project chose to include.

## Redactions

Internal cluster network addressing and one dead local credential were redacted for security — see [`REDACTIONS.md`](REDACTIONS.md). The set is otherwise faithful to what trained the model.
